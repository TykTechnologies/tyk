2. General Description

Requirements for Tyk API Gateway

2.1 Product Perspective

Tyk API Gateway is a requirements engineering tool designed for teams that need formal verification of their specifications. It occupies a unique position in the requirements ecosystem:

  • Compared to IBM DOORS / Jama Connect: Tyk API Gateway provides git-native storage (YAML files under version control) rather than a proprietary database, enabling standard code-review workflows and offline access.
  • Compared to NASA FRET: Tyk API Gateway adds persistent storage, traceability management, gap analysis, and document generation on top of FRETish compilation, operating as a complete requirements lifecycle tool rather than a formalization workbench.
  • Compared to Doorstop: Tyk API Gateway adds formal verification (realizability, consistency, vacuity checking via Kind2/JKind) and automated MC/DC test generation via the FLIP algorithm.

2.2 Product Functions

At a high level, Tyk API Gateway provides the following capabilities:

  • Compile: Parse FRETish requirements (288 semantic templates) into future-time and past-time LTL, and CoCoSpec Lustre contracts.
  • Verify: Check realizability, consistency, and vacuity of formalized requirements using Kind2 or JKind model checkers, with assurance levels A through E.
  • Analyze: Perform gap analysis across 7 categories (parent traceability, test coverage, implementation links, formalization completeness, review status, output determinism, assumption coverage).
  • Test: Generate MC/DC test cases from temporal logic formulas using the FLIP algorithm (Formal LTL-based Input Partitioning).
  • Generate: Produce NPR 7150-compliant SRS documents, traceability matrices, verification reports, and requirement summaries in HTML and Markdown formats.
  • Manage: Baseline requirements, track changes, compute diffs between baselines, and maintain full provenance records.

2.4 Constraints

  • Formal verification (realizability, consistency, vacuity) requires Kind2 or JKind to be installed and accessible on the system PATH.
  • The FRETish compiler supports the 288 semantic templates defined by NASA FRET; requirements outside this template space must use manual LTL or informal verification strategies.
  • Document generation currently supports HTML and Markdown output; PDF generation requires a future chromedp (headless Chromium) integration.
  • The tool is distributed as a single Go binary; the Go runtime is statically linked and not required at deployment time.

2.5 Assumptions and Dependencies

  • The FRETish semantic engine is based on the 288 NASA templates from the FRET project (Giannakopoulou et al., NASA Ames).
  • The ANTLR4-compatible grammar defines the FRETish language syntax; changes to the grammar may affect requirement parsing.
  • Kind2 and JKind are external SMT-based model checkers maintained independently; solver compatibility is tested against Kind2 v1.x and JKind v4.x.
  • Git is assumed to be available for repository operations (baseline creation, change tracking).

3. Specific Requirements

3.1 policy

The policy package implements Tyk API Gateway's policy merging engine. It takes a session (API key) referencing one or more policies and applies those policies to the session, producing merged access rights, rate limits, quotas, tags, metadata, and complexity limits.

ARCHITECTURE

The Service struct holds a policy store and logger. Apply() iterates over policy IDs, looks up each policy, validates org ownership, and dispatches to either applyPerAPI() or applyPartitions() based on the PerAPI flag. ClearSession() resets session values before application. ApplyRateLimits() implements the highest-rate-wins merge. ApplyEndpointLevelLimits() merges per-endpoint rate limits. MergeAllowedURLs() merges per-URL ACLs. The Store and StoreMap provide policy lookup.

DESIGN DECISIONS

(1) Highest value wins for rate limits and quotas, ensuring the most permissive policy governs. (2) Per-API mode and partition mode are mutually exclusive. (3) Missing policies are skipped when multiple policies exist but cause errors for single-policy sessions. (4) Org ID mismatch is a hard error (security boundary). (5) Session clearing happens before application to prevent stale values from surviving.
